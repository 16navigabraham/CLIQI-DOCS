# Background Jobs

All background jobs use the **SafeInterval** pattern — a crash-resilient wrapper around `setInterval` that prevents overlapping executions, catches errors, and sends admin alerts via Telegram.

## SafeInterval Features

- **Crash-resilient:** If a job throws, the error is caught and logged — the interval continues
- **Overlap-safe:** If a previous run hasn't finished, the next one is skipped
- **Admin alerts:** All errors automatically sent to admin Telegram DM (rate-limited: 1/min per error type)
- **Idle-aware:** Jobs gate on `activityGate.isActive()` to skip when no users are connected (server sleep mode)

## Job Details

### Price Push (30s)

For each connected WebSocket user:
1. Fetch their wallet token holdings (Alchemy for Base, RPC for Solana)
2. Get live prices via the price oracle (GeckoTerminal → DEXScreener → stale cache → DEX aggregator quote fallback)
3. Push `prices:update` event with all token prices + values
4. Skips users with no active WebSocket connection

### Stuck Trade Resolver (5m)

Catches trades that fell through the cracks:
1. Query database for trades in `PENDING` or `SUBMITTED` status > 10 minutes old
2. For each stuck trade:
   - **Solana:** Query transaction status via RPC `getTransaction()`
   - **Base:** Query transaction receipt via `provider.getTransactionReceipt()`
3. Update trade status to `CONFIRMED` or `FAILED` based on on-chain result
4. Notify user via WebSocket if status changed

### Premium Expiration (1h)

Downgrades users whose premium subscription has expired:
1. Find users where `isPremium = true` AND `premiumUntil < now()`
2. Set `isPremium = false`
3. Send Telegram notification: "Your premium has expired"
4. Auto-sell monitoring stops for these users on next cycle

### Auto-Sell Monitor (30s) — Premium Only

Monitors portfolio positions against take-profit and stop-loss thresholds. Processes all users in parallel for fast cycle times (~1.3–1.5s).

**Architecture:**
1. Find all premium users with `autoSellEnabled = true`
2. Process all users in parallel via `Promise.allSettled` (not sequential)
3. For each user, fetch token balances using `fetchTokenBalances` — targeted per-chain RPC calls instead of full wallet scan
4. Fetch current token prices via price oracle
5. Check each position against thresholds:
   - **Take profit:** Triggers at the exact TP threshold (no slippage buffer) — e.g., 50% TP triggers at +50%
   - **Stop loss:** Triggers early with a slippage buffer — e.g., -17% SL with 3% buffer triggers at -14%, ensuring the sell executes before the actual threshold is breached
6. **Grace period:** Tokens bought within the last 3 minutes are skipped to prevent false stop-loss triggers on newly purchased tokens with volatile initial pricing
7. If triggered, execute a sell-all trade via the trade orchestrator
8. Send `auto-sell:executed` or `auto-sell:failed` WebSocket event
9. Diagnostic logging for every position check (token, PnL%, thresholds, prices)

**Performance optimisations:**
- **Parallel user processing:** All users checked simultaneously, not sequentially
- **Balance cache:** 120-second TTL per user, invalidated on sell execution
- **`fetchTokenBalances`:** Groups tokens by chain, one RPC call per chain (Solana `getTokenAccountsByOwner` + Alchemy `getTokenBalances` for Base), with direct `balanceOf` fallback for EVM tokens not in Alchemy index
- **Price cache warmup:** Native prices (SOL, ETH) pre-fetched on server startup to avoid null prices after cold restart

### P&L Monitor (90s) — Premium Only

Sends Telegram DM notifications when a user's position P&L changes significantly:

1. Find all premium users with Telegram connected and notifications enabled
2. For each user, call `/api/trades/portfolio` (same endpoint as frontend) using a short-lived internal JWT
3. For each open position, check:
   - **Milestone crossing:** P&L crosses ±20%, ±50%, ±100%, ±200%, ±500%, or -75%
   - **Significant swing:** P&L moved >10% since last notification
4. If triggered, send Telegram DM with current value, P&L %, and Sell button
5. Store last-notified P&L in Redis with 6h TTL (stops notifying if position stays flat)
6. Detect closed positions: if a previously-tracked position is now `status: 'closed'`, send a "Position Closed" notification with final P&L and clean up the Redis key

Redis keys: `pnl_notif:{userId}:{chain}:{tokenAddress}` (6h TTL)

### Queue Stats (1m)

Logs trade queue health metrics when the queue is not empty:
- Current queue length
- Active (executing) trade count
- Pending trade count
- Average wait time

## Activity Gate (Server Sleep Mode)

To conserve resources on the 512 MB Render instance:

- `activityGate.isActive()` returns `true` only when at least one WebSocket client is connected
- When no users are connected, all background jobs skip their execution
- The server effectively "sleeps" until a user connects
- This prevents unnecessary API calls, RPC queries, and database load during idle periods
